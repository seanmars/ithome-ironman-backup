# Day 16  專案最後階段交付大禮包

> 原文連結: https://ithelp.ithome.com.tw/articles/10380996

![day16](https://hackmd.io/_uploads/HykxZ4E9xl.png)

> 文件不是為了方便使用，而是為了讓客人覺得「你想得比他還周到」。

### 為什麼需要「交付大禮包」？

在專案收尾階段，很多工程師只想趕快把東西交出去，認為「功能完成、Bug 修掉」就代表結束了。  
但對牛郎工程師來說，最後這一步才是關鍵時刻，因為它決定了客人對整個專案的最終印象。

這時候，你需要準備一份 交付大禮包。  
它可能包含：

* 文件清單
* 體貼的備註
* 以防萬一的提醒與注意事項

這些文件未必會被翻開，但它們的存在價值在於：

* 體現細心：讓客人覺得你比他更周到。
* 建立安全感：即使專案交付後，也不會覺得被丟下。
* 放大專業形象：留下「專業、貼心」的收尾印象。

很多時候，客人真正想要的不是「文件有多實用」，而是那份「我被照顧得很周全」的感受。

### 具體做法

* 提供清楚的文件目錄與摘要
  + 讓客人能一眼找到需要的資訊，不必東翻西找。
  + 可以用 README.md 或一張 PDF 目錄頁，附上簡短說明。
* 附上貼心備註
  + 在每個重要文件或步驟中，額外標註：「此步驟可能需要額外注意」、「若發生問題可參考此處」。
  + 就算客人不是專業人士，也能感受到被保護的安心。
* 加入「以防萬一」資訊
  + 例如：「若遇特殊情況，請參考附件中的緊急聯絡方式」。
  + 這些內容未必真的會用上，但客人會覺得你想得比他還多。
* 保持簡潔可讀性
  + 記得：大禮包不是資料傾倒，而是「貼心整理」。
  + 少而精，比堆一堆客人看不懂的技術細節更能加分。

### 範例

* 「專案已經完成，這裡是我為您整理的所有文件清單，並附上了一些注意事項，希望對您後續追蹤或維護有幫助。」
* 「為了確保萬無一失，我特別準備了幾個額外注意事項和聯絡資訊，方便您日後需要時能快速參考。」

這些話，不只是交代工作，而是在傳遞一個訊號：  
「我已經替你想到未來的可能情況，請放心交給我。」

### 職場延伸

「交付大禮包」的概念，不只適用於專案收尾，也可以運用在：

* 會議之後：整理會議重點＋行動清單，讓大家覺得你很可靠。
* 需求討論後：做一份快速紀錄，附上「下一步建議」。
* 交接工作時：除了交接文件，再加上一些「小提醒」或「常見陷阱」。

這些動作都不是技術層面的加分，而是讓別人覺得和你合作很輕鬆。

### 長期效果

* 你在客人心中會被標記為「可靠、專業、體貼」。
* 即使專案已經收尾，客人仍然會帶著正面印象。
* 下次合作時，客人會優先想到你，因為你「交付的不只是專案，而是一份安心」。
